# Fennara MCP Guidelines

These guidelines are generated by the Fennara installer. They are intentionally detailed because MCP clients and coding agents need explicit rules to work safely inside Godot projects.

## Read This First

Before starting any task that touches Godot-specific files or behavior, read this file and follow it.

Godot-specific work includes:

- `.gd` scripts
- `.cs` scripts
- `.gdshader` shaders
- `.tscn` scenes
- `.tres` and `.res` resources
- `project.godot`
- nodes, scenes, resources, exported properties, signals, groups, autoloads, input map, rendering settings, physics settings, and project settings
- UI layout, Control nodes, themes, cameras, animation, shaders, materials, gameplay, editor/addon behavior, and Fennara behavior

Use Fennara MCP whenever it can inspect, edit, or validate the Godot state directly. Do not rely only on raw file reads, generic filesystem search, or memory for Godot work.

## Required Connection Checks

Fennara MCP tools require:

- Godot open with this project loaded
- Godot 4.x. Fennara does not support Godot 3.x.
- the Fennara addon installed in this project
- the target project selected as the MCP target in the Fennara dock

Use `fennara_status` first when any of these are uncertain:

- whether the daemon is reachable
- whether Godot is connected
- which project is the active MCP target
- which Fennara MCP tools are available

If more than one Godot project is open, the active MCP target controls which project receives Fennara MCP calls. In Godot, use the Fennara dock's project target control for the desired project.

## Cursor MCP Identifier

In Cursor settings and the user's `mcp.json`, the Fennara MCP server is normally named `fennara`.

Cursor may expose the same user-added MCP server to agents or cached project MCP metadata with the internal identifier `user-fennara`. Treat `fennara` and `user-fennara` as the same server, not two different Fennara installs:

- `fennara` is the human-facing name in Cursor Settings and `mcp.json`.
- `user-fennara` is Cursor's internal identifier for that user-added MCP server.

When using MCP tools programmatically, if `fennara` is not found, also check for `user-fennara`. Do not tell the user to look for `user-fennara` in Cursor settings; that name is internal plumbing.

## Core Rule

Inspect first. Edit second. Validate third.

For Godot work:

- If inspecting Godot state, use Fennara MCP inspection tools.
- If changing Godot state, use Fennara MCP editing tools.
- If validating Godot state, use Fennara MCP diagnostics, validation, or screenshot tools.

Do not inspect or modify Godot scenes, resources, scripts, or project settings solely through generic text editing when a Fennara MCP tool exists for the task.

Do not use generic file write/edit tools for `.gd`, `.cs`, `.gdshader`, `.tscn`, `.tres`, `.res`, or `project.godot` changes when a Fennara MCP tool can perform the edit. Generic app file tools are for reading, searching, diffs, and non-Godot files. Godot-aware edits must go through Fennara MCP so diagnostics, scene validation, resource serialization, and editor state stay correct.

Never hand-write or directly patch `.tscn`, `.tres`, or `.res` files as plain text. Those files are Godot-serialized resources; edit them through `run_scene_edit_script`, `save_custom_resource`, `project_settings`, or another relevant Fennara MCP tool.

## Tool Schema Rule

Before choosing a Fennara tool, read the available tool schemas and descriptions exposed by your MCP client. Treat those schemas as the source of truth for:

- exact tool names
- required arguments
- optional arguments
- allowed limits
- result shape
- validation behavior
- any tool-specific workflow notes

The guidance in this file explains when to use tools. The live tool schema explains exactly how to call them.

## Deferred Tool Search Rule

Some MCP clients expose tools through a deferred search tool such as `tool_search` instead of loading every MCP tool immediately.

When searching for Fennara MCP tools through a tool-search system, search for all known Fennara tool names together and use a high limit when the search API supports one.

Use this query:

```text
fennara_status read_file file_ops write_or_update_file run_scene_edit_script get_scene_tree save_custom_resource script_diagnostics screenshot_scene get_node_properties get_class_info validate_scene project_settings runtime_session runtime_script scrape_editor
```

If the search tool has a `limit` option, set it to `20` or higher. Do this even when you only need one specific tool, because low result limits can bury relevant Fennara tools below the cutoff.

After the search returns, read the returned tool schemas before calling any tool. If `fennara_status` reports that a tool exists but the callable schema is not exposed yet, retry tool discovery with the exact tool name, the full query above, and a high limit.

## Fennara MCP Does Not Replace Normal Repository Navigation

Fennara MCP includes scoped `read_file` and `file_ops` helpers for cases where
Godot-side path normalization, image handling, or bundled ripgrep search is
useful.

Use your MCP app's own file tools for broad ordinary file reading, diffs,
repository navigation, and non-Godot text inspection.

Use Fennara MCP for Godot-aware work: inspecting scene trees, node properties, native Godot APIs, runtime errors, editor debugger snapshots, diagnostics, validation, screenshots, and project settings.

## How To Understand Code And Assets

Use the narrowest tool that can answer the question.

- Need a specific known file or a few known files: use your MCP app's file-reading tools.
- Searching for a function, class, variable, signal, setting, or returned log text: use your MCP app's text search tools.
- Finding files by pattern: use your MCP app's file search/glob tools.
- Need scene structure: use `get_scene_tree`.
- Need changed node properties, attached scripts, exported variables, connections, resources, or animation data: use `get_node_properties`.
- Need native Godot API details before scene/resource edits: use `get_class_info`.
- For `.tscn` files, do not rely on raw file reads. Use `get_scene_tree`, `get_node_properties`, `validate_scene`, and targeted text search only when exact serialized text is needed.
- Use forward-slash Godot paths (`res://...` and `user://...`) for compatibility across Windows, macOS, and Linux.

## Large Scene And Timeout Discipline

Some MCP clients can be configured to wait several minutes for Fennara tool calls so valid Godot/editor operations on large projects can finish. Do not treat that longer timeout as a normal budget or permission to brute-force huge scenes.

## Tool Call Logs And Timeout Recovery

The Godot addon writes searchable MCP tool lifecycle logs under `user://.fennara/tool_logs/<session_id>/`.

- `calls.jsonl` is an append-only event log. Search it by `request_id`, tool name, scene path, script path, or other input text.
- Simple tool calls store their model-facing result as `results/tool_<n>_<tool_name>.json`.
- Artifact-producing tools store their model-facing result as `results/tool_<n>_<tool_name>/result.json` and place logs, screenshots, command files, and other payloads beside that file.
- A daemon-managed runtime session owns one artifact folder, for example `results/tool_10_runtime_session/`. That folder contains `runtime_session.log`, command files, per-script status files, and `captures/`. Later `runtime_script` calls are small receipt JSON files that point back to the runtime session artifact folder instead of creating separate screenshot locations.
- Important events are `received`, `started`, `completed`, and `failed`.
- The `received` event includes the original tool input. Final events include `result_path`, `artifact_path`, and `duration_ms`.

If a Fennara MCP tool times out or the app stops waiting before showing the result, inspect the latest matching events in `calls.jsonl` before retrying. If the latest event for that `request_id` is `completed` or `failed`, open the linked result JSON and use that result. If the latest event is `started` with no later final event, the Godot-side tool may still be running; wait a few minutes and check the log again. If it remains `started` after the supported tool window, stop waiting and retry with a smaller scope, narrower paths, or a less expensive script. Do not blindly retry the same broad call while the previous one may still be running.

For large root/world scenes, avoid broad full-scene scans unless the task truly requires them. Prefer targeted inspection first:

- use `get_scene_tree` to find candidate paths
- use `get_node_properties` for known nodes/resources
- use normal file search/read tools for text-only questions
- use `run_scene_edit_script` only when native Godot object access is needed

When a broad `run_scene_edit_script` scan is unavoidable, start with a small probe, log concise progress checkpoints and final counts with `ctx.log(...)`, skip irrelevant subtrees early, cache expensive per-node calculations, cap sample output, and avoid recomputing parent/category/AABB/resource data in multiple loops.

If a Fennara tool call takes unusually long or times out, do not immediately retry the same broad scan. Narrow the scene/path/filter, patch the returned `script_path` when available, add checkpoint logs, or split the work into smaller passes.

## Runtime Sessions

`runtime_session` starts, checks, or stops one daemon-managed windowed Godot runtime session. `start` first runs scene-execution gates: C# projects run `dotnet build`, the requested scene gets structural `validate_scene` preflight without a headless runtime run, and Fennara checks autoload plus scene-attached scripts with targeted diagnostics. If those fail, Fennara does not open the scene. Fennara currently allows one managed runtime session globally across all connected Godot editors; if another runtime session is already running, `start` is blocked. Call `runtime_session.status` to inspect it or `runtime_session.stop` to close it before starting another scene. If it starts, the returned `runtime_session.log` is the source of truth for scene startup output, raw Godot stdout/stderr, runtime errors, `FENNARA_SCRIPT_*` markers, `ctx.log(...)` messages, captures, and completion/failure events. The log is written live as the scene and runtime scripts run, so during one still-running session you can inspect or search `runtime_session.log` between `runtime_script` calls before deciding the next script. Read that log after `runtime_session.start`, after every `runtime_script`, and when checking `runtime_session.status`.

`runtime_script` sends one short `RefCounted` GDScript inspector/input-driver probe into that running scene through Fennara's runtime helper. A script may complete without closing the scene; that is the normal incremental workflow when you want to submit another runtime_script against the same live scene. Treat the `runtime_script` tool result as an operational receipt, not proof that the scene had no runtime errors. Close the scene when finished with `runtime_session.stop` or with a final script that calls `ctx.close_scene()`.

Write runtime scripts so the shared runtime log stays token-efficient. Use `ctx.log(...)` only for compact milestones, sampled state, and final summaries; do not log every frame, loop iteration, action press/release, mouse move, or full node dump. In loops, accumulate counters, min/max values, small sample arrays, and goal status, then emit one concise summary log plus a few named captures.

Runtime controls are intentionally low-level and game-agnostic. For unknown gameplay mechanics, do not assume common genres, objectives, movement semantics, or success conditions. Treat all gameplay semantics as project-local until observed. Agents must inspect the project's real controls and game code, then compose primitive runtime actions for that specific game. Detailed runtime_script syntax, ctx methods, strict typing requirements, and input-driving patterns live in the runtime_script tool schema.

First run small inspection/probing scripts to discover the live control model and state signals: inspect InputMap action names, controller/player scripts, scene tree, UI, signals, animations, transforms, resources, and current runtime state before driving. Test one primitive input at a time for a short bounded duration, release actions, wait at least one frame, then observe what changed. Infer action effects from measured state deltas, not from action names.

Use adaptive observe -> experiment -> infer -> act -> verify loops. Keep experiments bounded by time or iteration count, re-fetch live nodes/state inside loops, and log compact summaries of attempted inputs, observed deltas, inferred effects, chosen next steps, and verified outcomes. If no reliable verification signal is found, report uncertainty and gather more observations instead of claiming success.

Await yielding runtime ctx helpers in scripts: `await ctx.wait(...)`, `await ctx.capture(...)`, `await ctx.tap_action(...)`, `await ctx.action(...)`, `await ctx.action_sequence(...)`, `await ctx.click_at(...)`, and `await ctx.click_button(...)`. Without `await`, a script may continue immediately and produce false observations. Synchronous helpers such as `ctx.log(...)`, `ctx.press_action(...)`, `ctx.release_action(...)`, `ctx.set_mouse_position(...)`, `ctx.release_all_actions()`, and getters do not need await.

Do not derive success from intent. Pressed input, proximity to a target, clicked coordinates, elapsed time, or suggestive action names are not verification. Verify success from actual runtime state after the game has processed at least one frame, such as a property/counter/UI text change, signal firing, node disappearing or becoming inactive/hidden/disabled, animation/state/scene transition, resource/object count change, or another project-specific state change discovered through inspection.

Do not encode genre-specific helpers or assumptions such as navigation, collecting, attacking, jumping, talking, inventory use, pathing, quests, combat, economy, or UI flow. Fennara tools and schemas must remain primitive and game-agnostic: agents should compose low-level inspection, input, timing, capture, and validation primitives for the specific project they are observing.

Avoid these runtime_script anti-patterns:

- holding one guessed action for several seconds without recomputing from live state
- assuming InputMap action names map to world directions or common gameplay verbs
- incrementing success counters from proximity, attempted input, or elapsed time
- keeping stale node references across scene reloads, respawns, or object replacement
- logging every frame instead of concise samples and summaries
- calling yielding ctx helpers such as wait(), capture(), action(), tap_action(), action_sequence(), click_at(), or click_button() without await
- leaving actions pressed after a loop, failure, timeout, or early return

## Editor Scraping

`scrape_editor` scrapes a narrow part of the currently open Godot editor UI. For now, call it only as `scrape_editor` with `target: "debugger"` to read the current editor debugger/errors tree snapshot. It does not start or stop scenes, run validation, inspect `runtime_session.log`, or take screenshots. The result follows the same token-efficient spirit as runtime session logging: repeated issues are grouped with counts, issue/detail lines are capped, and raw editor UI spam is omitted from the model-facing response. Use it only when the user says they manually ran a scene in Godot/the editor and got a runtime error, or when they explicitly ask what the editor debugger currently shows. Do not use it for scenes you started with `runtime_session`; use `runtime_session.status` and inspect/search the live `runtime_session.log` instead.

## Scene Tree Inspection

Use `get_scene_tree` before referring to node paths or editing `.tscn` scenes.

Use it when:

- you need to know the actual node hierarchy
- you need node paths for scripts or scene edits
- you are about to modify UI, cameras, player nodes, enemies, managers, animation players, or resources owned by a scene
- the user refers to a node by a human name but the exact path is unknown

Do not guess node paths. Read the scene tree first.

## Node Properties

Use `get_node_properties` before changing existing nodes or scene-owned resources.

Use it when:

- changing Control layout, anchors, offsets, size flags, theme overrides, text, visibility, process settings, collision shapes, scripts, exported properties, materials, or resource references
- diagnosing why a scene behaves differently than expected
- inspecting current node configuration before applying an edit

Do not assume current node or resource properties. Inspect them first.

## Node Naming

Always give created Godot nodes explicit, meaningful names unless there is a very strong reason not to.

Godot auto-generates names such as `@Label@21109`, `@HBoxContainer@21112`, or `@ProgressBar@21110` when a node is created without an explicit `name`. These names mean "this is a node of that class, but nobody named it." They are usually harmless at runtime, but they make the scene tree ugly, harder to inspect, harder to diff, and harder for scripts or humans to maintain.

This behavior is discouraged for AI-authored scenes. When creating nodes through `run_scene_edit_script` or any Godot scene-editing workflow, set names immediately for important, visible, repeated, scripted, animated, inspected, or maintained nodes.

Use clear names such as:

- `StrengthRow`
- `StrengthLabel`
- `StrengthBar`
- `StrengthValue`
- `DexterityRow`
- `DexterityBar`

Generated or repeated purely decorative children may use auto names only when there is a very strong reason and no script, animation, signal, test, inspection workflow, or future maintenance depends on them. When in doubt, name the node.

## Godot API Lookup

Use `get_class_info` before relying on native Godot APIs.

Use it for:

- class inheritance
- methods
- properties
- signals
- enums
- constants
- constructor assumptions
- method signatures
- built-in resource/node capabilities

Do not invent Godot API names from memory when the class info tool can confirm them.

## File Edits

Use `write_or_update_file` for `.gd`, `.cs`, and `.gdshader` edits and other Godot project text edits when a Fennara MCP edit is appropriate.

Important behavior:

- `write_or_update_file` automatically runs diagnostics for `.gd`, `.cs`, and `.gdshader` files.
- For `.gdshader` edits, `write_or_update_file` also scans `.tscn` and `.tres` files that reference the shader and reserializes those owners through Godot when possible, so stale embedded `ShaderMaterial` data can be rewritten.
- Check `reserialized_resources`, `reserialize_warnings`, and `reserialize_skipped` in `.gdshader` edit results. Skipped owners may still need a targeted scene/resource save.
- If it reports diagnostics, treat them as part of the tool result and fix them before claiming the script edit is complete.
- For other file types, use appropriate follow-up validation when the file affects Godot behavior.
- Do not use the MCP app's generic write/edit tool for `.gd`, `.cs`, or `.gdshader` files. Use `write_or_update_file` so diagnostics run automatically.
- Do not use `write_or_update_file` or generic text editing for `.tscn`, `.tres`, or `.res` scene/resource surgery. Use `run_scene_edit_script` or the specific structured Fennara tool instead.

For ordinary text reading, use the MCP app's own file tools.

## Scene And Resource Edits

Use `run_scene_edit_script` for procedural scene or resource edits through Godot's object model.

Use it instead of raw `.tscn`, `.tres`, or `.res` surgery when editing:

- node hierarchy
- node properties
- scene-owned resources
- Control layout
- collision shapes
- cameras
- AnimationPlayer setup
- signals and groups
- scripts attached to nodes
- materials and visual resources

Important behavior:

- `run_scene_edit_script` automatically runs script diagnostics after edits.
- `run_scene_edit_script` automatically runs scene validation after edits.
- Treat diagnostics and validation output as part of the edit result.
- Fix reported issues before claiming the scene/resource edit is complete.
- Do not directly write `.tscn` files with generic file tools. If a scene must change, use `run_scene_edit_script` so Godot owns serialization and validation.
- For inspection-only scripts, use `ctx.log(...)` and do not call `ctx.mark_modified()` or mutating helpers such as `ctx.own(...)`, `ctx.instance_scene(...)`, `ctx.remove_node(...)`, or `ctx.clear_children(...)`. Existing scenes with `modified=false` are not saved, so read-only inspection should report `scene_saved=false` and should not rewrite the `.tscn`.

Inherited scenes:

- Some scenes inherit another scene at the root, shown in `.tscn` text as `[node name="SomeScene" instance=ExtResource(...)]`.
- `run_scene_edit_script` tries to preserve inherited roots by loading inherited scenes in inherited edit mode and saving valid overrides/additions instead of flattening the base scene into local nodes.
- If Godot would flatten the inherited root during save, the tool refuses to keep that save, restores the original scene file, and returns `success=false`, `scene_saved=false`, `inherited_root_scene=true`, and an error explaining that preserving the inherited root failed.
- When that happens, do not assume your edit applied and do not patch the `.tscn` text by hand. Patch the returned `script_path` or choose a narrower scene edit, then rerun.
- For inherited scenes, prefer targeted property/resource overrides on existing nodes. Avoid replacing the root, clearing broad inherited subtrees, recursively owning inherited children, or rebuilding inherited structure.

Temporary edit scripts:

- Inline `run_scene_edit_script` code may be saved by Fennara under `.fennara` automatically; that is tool-managed scratch space.
- If you intentionally create a reusable or manual scene-edit script inside the Godot project, save it under `res://temp/`.
- Do not put one-off scene-edit scripts in normal runtime folders such as `res://scripts/`, `res://scenes/`, or feature directories unless the user explicitly wants to keep that authoring script as project source.
- Scene-edit scripts are editor/authoring helpers, not gameplay scripts. Do not attach them to runtime nodes.

## Script Diagnostics

Use `script_diagnostics` when:

- investigating script errors
- editing GDScript, C#, or Godot shader code outside a tool path that already ran diagnostics
- checking multiple scripts
- checking project-level GDScript, C#, and shader health
- confirming a suspected parse/type/reference issue
- errors only appear when opening scenes in Godot, such as invalid calls emitted by attached scripts while a scene is constructed

Targeted `script_diagnostics` calls support `.gd`, `.cs`, and `.gdshader` files. `scan_project: true` scans `.gd`, `.cs`, and `.gdshader` files under `res://`.

For requested `.gd` and `.cs` files, `script_diagnostics` also loads and instantiates project `.tscn` scenes in memory with Godot logger capture enabled. Script-related scene-load errors are attached to the matching script file with `source: "scene_load"` and `scene_path`, so the result says which scene triggered the script error. This is an in-memory diagnostic pass, not opening the editor UI or running the game scene.

If `write_or_update_file` edited a `.gd`, `.cs`, or `.gdshader` file, it already ran diagnostics. For `.gdshader`, it also attempted to reserialize referencing `.tscn` and `.tres` owners. You may still run `script_diagnostics` separately for broader checks or if the task requires extra confidence.

Do not consider a GDScript, C#, or Godot shader change complete until diagnostics have been checked.

## Scene Validation

Use `validate_scene` when:

- investigating scene/resource issues
- checking scenes after raw file edits
- validating scene integrity
- confirming node/resource references after changes
- checking scenes not covered by a previous automatic validation

`validate_scene` accepts 1-10 scene paths. It runs structural scene checks first. For scenes with zero structural errors, it also runs each scene headlessly for exactly 3 seconds through the local daemon using 3 memory-throttled workers, then returns each scene's compacted runtime output inline and lists where the full result JSON and raw runtime logs were saved. It does not open scenes in the editor or scrape the editor Output panel.

If `run_scene_edit_script` edited a scene/resource, it already ran scene validation after the edit. You may still run `validate_scene` separately for additional scenes or if the task requires broader coverage.

Do not consider a scene change complete until validation has been checked.

## Screenshots

Use `screenshot_scene` when visual correctness matters.

Use it heavily for Godot work whenever it can provide useful visual confirmation, especially after scene, camera, UI, material, shader, animation, or layout edits. It is not mandatory after every small tool call, but it is one of the best ways to catch mistakes that diagnostics and scene validation cannot see.

Use it for:

- UI layout
- Control anchoring and spacing
- camera framing
- rendering issues
- material/shader work
- animation-visible state
- visual regressions
- anything the user would judge by looking at the game/editor output

Do not rely only on text inspection for visual tasks.

## Project Settings

Use `project_settings` for:

- `project.godot`
- input map
- autoloads
- display/window settings
- rendering settings
- physics settings
- application metadata
- addon/project configuration stored in project settings

Prefer this over manual `project.godot` edits when possible.

## Fennara Autoloads And Addon Scripts

Fennara may install required GDScript support files and global autoload entries in the user's Godot project so the addon and MCP bridge can work correctly.

These Fennara-added scripts and autoloads are not noise, dead code, or cleanup targets. Do not remove, rename, disable, or "simplify" Fennara autoload entries from Project Settings unless the user explicitly asks to uninstall or repair Fennara itself.

If a Fennara autoload or addon script appears related to a warning, first inspect why it exists and report the finding. Prefer fixing the actual issue or asking the user before touching Fennara infrastructure.

## Custom Resources

Use `save_custom_resource` for `.tres` resources when a structured Godot resource save is needed.

Use it when working with custom Resource classes, reusable data assets, materials, or configuration assets that should be written through Godot's resource system.

## Recommended Workflows

### Investigating Existing Behavior

1. Read likely files with the MCP app's file tools.
2. Use `get_scene_tree` for relevant scenes.
3. Use `get_node_properties` for relevant nodes/resources.
4. Use `get_class_info` for Godot APIs before making assumptions.
5. Summarize findings with file paths, scene paths, node paths, and tool evidence.

### Editing GDScript Or C#

1. Read the file with the MCP app's file tools.
2. Use `get_class_info` for any uncertain Godot API.
3. Edit `.gd`, `.cs`, and `.gdshader` files with `write_or_update_file`. Do not use the MCP app's normal write/edit tool for GDScript, C#, or Godot shader files.
4. Ensure diagnostics are checked. `write_or_update_file` does this automatically for `.gd`, `.cs`, and `.gdshader` files.
5. Fix diagnostics before finishing.

### Editing Scenes

1. Use `get_scene_tree`.
2. Use `get_node_properties` for nodes/resources you will touch.
3. Use `get_class_info` for relevant node/resource APIs.
4. Edit scenes with `run_scene_edit_script` or another structured Fennara scene/resource tool. Do not directly write `.tscn` text.
5. Check the diagnostics and validation returned by `run_scene_edit_script`.
6. Use `screenshot_scene` when visual correctness matters.

### Editing Project Settings

1. Use `project_settings` to inspect current values.
2. Change settings through `project_settings` when supported.
3. Validate affected scenes/scripts if the setting impacts runtime behavior.

### Investigating Runtime Errors

1. Use `runtime_session.start` for a fresh scene run, then read the returned `runtime_session.log`.
2. If a session is already running, use `runtime_session.status` and inspect/search the same live `runtime_session.log`.
3. If the scene was started manually in the Godot editor or the user asks what the editor debugger currently shows, use `scrape_editor` with `target: "debugger"`.
4. Use stack frames and error lines from the runtime log or debugger snapshot to read the relevant scripts directly.
5. Use `script_diagnostics` only if the runtime error suggests script/shader parse/type/reference issues or after editing `.gd`, `.cs`, or `.gdshader` files.
6. Fix the smallest relevant issue and validate with a fresh runtime session, another runtime script, `scrape_editor`, or suitable diagnostics.

### Visual/UI Work

1. Use `get_scene_tree` for the scene.
2. Use `get_node_properties` for relevant Control nodes, containers, themes, labels, buttons, cameras, and resources.
3. Use `get_class_info` for Control/container APIs when uncertain.
4. Edit through `run_scene_edit_script` where possible.
5. Use `screenshot_scene` to verify the result.
6. Iterate until the screenshot matches the user request.

## Completion Rules

A Godot task is not complete just because files were edited.

Before finishing:

- Mention which Fennara MCP tools were used.
- Mention any diagnostics or validation results.
- If visual work was involved, mention whether a screenshot was checked.
- If a relevant Fennara tool could not be used, say why.

For GDScript, C#, or Godot shaders:

- Diagnostics must be checked.
- `write_or_update_file` automatically checks diagnostics for `.gd`, `.cs`, and `.gdshader` files.
- `.gdshader` edits also report referencing owner reserialization fields: `reserialized_resources`, `reserialize_warnings`, and `reserialize_skipped`.

For scene/resource edits:

- Scene validation must be checked.
- `run_scene_edit_script` automatically checks diagnostics and scene validation after edits.

For visual work:

- A screenshot should be checked unless the user explicitly says not to.

## Tool Discipline

- Do not invent tool results.
- Do not claim unavailable tools.
- Do not reveal or describe internal tool schemas beyond normal user-facing explanation.
- Do not use broad file reads when targeted search, scene inspection, diagnostics, or runtime probing is enough.
- Prefer fewer, deeper Godot-aware observations over generic guesses.
- If the same tool call fails twice in the same way, stop retrying and explain what happened.

## Failure States

If a Fennara MCP tool fails because the project is not connected, tell the user to open the Godot project with the Fennara addon installed.

If a Fennara MCP tool fails because the wrong project is active, tell the user to use the Fennara dock to set the desired project as the MCP target.

If a tool fails for another reason, report the tool name, the high-level error, and the next concrete step.

## Bias Toward Minimal Changes

Keep changes minimal and scoped to the user request.

Do not create broad refactors, new abstractions, or unrelated rewrites unless the task requires them.

Prefer the existing project style and Godot patterns.

Use Fennara MCP to reduce guessing, not to overbuild.
